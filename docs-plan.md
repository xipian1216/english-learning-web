# docs-plan.md

## 1. 目的

本文件是一份用户可以手动提供给你的前端文档构建与维护指南。

当用户希望你创建、重建、整理或更新前端项目 `docs/` 目录时，你应该将本文件作为任务上下文使用。你需要根据本文件分析项目，并生成或维护适合前端开发、测试、设计协作和大模型接手的项目文档。

本文件不要求在每次开发对话中自动加载；只有当用户明确提供、引用或要求进行文档构建工作时，你才需要使用。

## 2. 文档体系目标

你需要帮助前端项目建立一个清晰、可维护、适合人类和大模型共同使用的项目知识库。

`docs/` 不应该把所有内容都写进少数几个大文档中。对于页面状态、交互流程、组件演进、接口对接、功能规划、视觉规范和发布状态等动态变化内容，你应该使用分层目录和小文档进行管理，避免单个文档过长、过时或难以维护。

`docs/` 应记录真实的前端项目事实，包括：

- 产品目标和用户场景。
- 当前版本和项目状态。
- 技术栈。
- 应用架构。
- 页面、路由和布局职责。
- 组件组织方式。
- 状态管理和数据流。
- API 客户端和后端契约。
- 表单、校验和错误展示规则。
- 视觉、响应式和可访问性约定。
- 开发流程。
- 编码规范。
- 测试策略。
- 构建、部署和环境配置。
- 功能规划。
- 已实现功能。
- 开发中功能。
- 待实现功能。
- 功能开发过程。
- 排障经验。
- 重要技术决策。

## 3. 标准 `docs/` 结构

你可以采用以下推荐结构：

```txt
docs/
  00-overview.md
  01-project-status.md
  02-architecture.md
  03-development-workflow.md
  04-coding-standards.md
  05-testing.md
  06-api-integration.md
  07-ui-ux-guidelines.md
  08-deployment.md
  09-troubleshooting.md
  10-decisions.md
  11-glossary.md

  features/
    README.md
    planned/
      README.md
      YYYY-MM-DD-feature-name.md
    in-progress/
      README.md
      YYYY-MM-DD-feature-name.md
    implemented/
      README.md
      YYYY-MM-DD-feature-name.md

  pages/
    README.md
    route-name.md

  components/
    README.md
    component-name.md

  releases/
    README.md
    current.md
    YYYY-MM-DD-version.md
```

你可以根据项目实际情况省略不相关文件，但一旦创建后，应尽量保持编号、目录职责和命名规则稳定。

## 4. 文档分层原则

你应该按内容变化频率和用途拆分文档：

1. **总览类文档**：记录项目长期稳定信息，例如产品目标、技术栈、架构、开发流程、编码规范。
2. **状态类文档**：记录项目当前状态，例如当前版本、当前阶段、开发重点、已知风险和下一步重点。
3. **功能类文档**：按功能拆分，分别记录计划中、开发中、已实现的功能。
4. **页面类文档**：按路由或页面拆分，记录页面职责、核心状态、交互流程和接口依赖。
5. **组件类文档**：记录可复用组件的职责、属性、事件、状态和使用边界。
6. **版本类文档**：记录版本状态、版本变更、发布历史和当前版本说明。
7. **决策类文档**：记录重要技术决策，不要混入功能开发日志。

你不应该把所有功能、页面、组件和开发过程都写入一个大文档。动态内容应该拆分成独立小文档，并通过索引文件进行汇总。

## 5. 顶层文档职责

### `00-overview.md`

你应该用它说明前端项目是什么，以及为什么存在。

应包含：

- 项目目的。
- 主要能力摘要。
- 目标用户或使用场景。
- 高层技术栈，例如 Vue、Vite、TypeScript、Pinia、Vue Router、Vitest 和 Playwright。
- 重要入口文件。
- 快速开始说明或链接。

不要在这里记录详细版本历史或完整功能清单；这些内容应放在 `01-project-status.md`、`features/` 或 `releases/` 中。

### `01-project-status.md`

你应该用它告诉新的对话：项目当前处于什么状态，下一步应该如何继续。

应包含：

- 当前项目阶段。
- 当前版本或最近版本。
- 当前开发重点。
- 已完成的关键能力摘要，并链接到 `features/implemented/`。
- 正在开发的功能摘要，并链接到 `features/in-progress/`。
- 准备实现的功能摘要，并链接到 `features/planned/`。
- 重要风险、阻塞点和待确认问题。
- 下一次开发会话开始前应该优先阅读的文档。

`01-project-status.md` 是面向新会话的入口状态文档。它不应记录过长的功能细节，而应通过链接指向具体功能文档、页面文档、组件文档和版本文档。

### `02-architecture.md`

你应该用它说明前端应用如何组织和运行。

应包含：

- 高层架构。
- 主要目录及其职责。
- 入口文件和应用挂载流程。
- 路由、布局、页面和组件关系。
- 状态管理模型。
- API 客户端和数据流。
- 表单校验、错误处理和鉴权边界。
- 与后端服务、认证服务或第三方服务的集成。
- 必要时添加图示。

### `03-development-workflow.md`

你应该用它说明开发者和大模型如何在本前端项目中工作。

应包含：

- 环境准备步骤。
- 本地开发命令。
- 常见开发流程。
- 分支、提交或 PR 约定，如适用。
- 文档更新流程。
- 发布流程，如适用。
- 功能文档在计划、开发中、已完成之间流转的规则。
- 页面文档和组件文档的新增、更新规则。

### `04-coding-standards.md`

你应该用它说明项目特定的编码约定。

应包含：

- Vue、TypeScript 和 Vite 约定。
- 组件命名、属性、事件和插槽规则。
- 页面、布局、组件、store、schema 和 API 文件组织规则。
- 路由命名和导航规则。
- Pinia 状态管理规则。
- 表单状态、校验和错误处理规则。
- 样式、响应式和可访问性规则。
- 依赖使用规则。

### `05-testing.md`

你应该用它说明测试策略和验证命令。

应包含：

- 测试类型，例如单元测试、组件测试、端到端测试、类型检查和构建验证。
- 测试目录结构。
- 如何运行测试。
- 完成任务前需要执行的检查。
- Mock API、路由和 store 的策略。
- 手动浏览器验证指南。
- 响应式、可访问性和关键交互的验证要求。

### `06-api-integration.md`

你应该用它说明前端与后端或第三方服务的契约。

适用时应包含：

- API 客户端组织方式。
- 请求约定。
- 响应约定。
- 错误格式和前端展示规则。
- 鉴权、授权和 token 生命周期预期。
- Schema 演进规则。
- Mock 数据和本地联调方式。

### `07-ui-ux-guidelines.md`

你应该用它说明视觉、交互和可访问性约定。

应包含：

- 页面布局规则。
- 组件视觉层级。
- 颜色、字体、间距和边框规则，如项目已有约定。
- 加载、空状态、错误状态和成功状态。
- 表单交互和校验反馈。
- 移动端和桌面端响应式规则。
- 键盘导航、焦点状态、语义标签和可访问性要求。

### `08-deployment.md`

你应该用它说明前端项目如何配置、构建、发布和部署。

应包含：

- 环境变量。
- 构建命令。
- 预览命令。
- 部署目标。
- 静态资源、基础路径和缓存策略。
- CI/CD 流程。
- 回滚或恢复说明。

### `09-troubleshooting.md`

你应该用它记录已知问题和解决方式。

应包含：

- 常见安装、构建、类型检查、Lint、测试和浏览器问题。
- 调试步骤。
- 环境相关问题。
- 网络请求、鉴权、路由和状态异常的诊断方式。
- 恢复操作手册。

### `10-decisions.md`

你应该用它记录重要技术决策。

使用轻量级 ADR 条目。

推荐格式：

```md
## YYYY-MM-DD - 决策标题

### 背景

当时存在什么问题、约束或取舍？

### 决策

最终决定了什么？

### 影响

该决策带来了哪些收益、风险和后续事项？
```

### `11-glossary.md`

你应该用它定义项目特定术语。

## 6. 功能文档职责

### `features/README.md`

你应该用它作为功能文档索引。

应包含：

- 功能文档分区说明。
- 计划中、开发中、已完成功能的链接。
- 功能文档迁移规则。

### `features/planned/YYYY-MM-DD-feature-name.md`

你应该用它记录尚未开始但已经有明确方向的功能。

应包含：

- 功能目标。
- 用户场景。
- 页面、组件、store 和 API 影响范围。
- 交互与状态草案。
- 待确认问题。
- 验收标准。

### `features/in-progress/YYYY-MM-DD-feature-name.md`

你应该用它记录正在开发的功能。

应包含：

- 当前实现状态。
- 已完成部分。
- 未完成部分。
- 关键文件和入口。
- 已知问题。
- 下一步任务。
- 测试和手动验证进度。

### `features/implemented/YYYY-MM-DD-feature-name.md`

你应该用它记录已实现功能的真实行为。

应包含：

- 功能摘要。
- 用户可见行为。
- 主要页面、组件、store 和 API。
- 重要边界条件。
- 测试覆盖情况。
- 后续可改进事项。

## 7. 页面和组件文档职责

### `pages/route-name.md`

当页面逻辑复杂、承载关键业务流程或涉及多个状态时，你应该为页面创建文档。

应包含：

- 路由路径和页面入口。
- 页面职责。
- 使用的布局和核心组件。
- 主要状态和数据来源。
- API 依赖。
- 用户交互流程。
- 加载、空状态、错误状态和权限状态。
- 测试和手动验证要点。

### `components/component-name.md`

当组件被多个页面复用、交互复杂或承担设计系统职责时，你应该为组件创建文档。

应包含：

- 组件职责。
- Props、事件、插槽和公开方法。
- 状态和副作用。
- 使用示例。
- 可访问性要求。
- 不适合使用该组件的场景。

## 8. 文档写作规则

你应该遵循以下写作规则：

- 只记录已经从代码、配置、测试或用户说明中确认的事实。
- 对尚未确认的信息使用“待确认”标注，不要写成确定事实。
- 文档应简洁、可扫描，优先使用列表、表格和链接。
- 每个文档只承担一个清晰职责。
- 动态状态应保留日期或版本线索。
- 修改功能、页面、组件、API、配置、测试或部署流程时，应同步更新相关文档。
- 当文档与代码冲突时，应指出冲突，并以实际代码和用户确认作为后续修改依据。

## 9. 文档维护流程

当用户要求构建或整理 `docs/` 时，你应该：

1. 阅读 `README.md`、`package.json`、源码目录、测试目录、构建配置和已有文档。
2. 识别当前前端技术栈、页面结构、路由、状态管理、API 客户端和测试方式。
3. 生成或更新顶层文档。
4. 为明确的功能补充 `features/` 文档。
5. 为复杂页面补充 `pages/` 文档。
6. 为复杂复用组件补充 `components/` 文档。
7. 更新 `01-project-status.md` 和 `releases/current.md`，确保新会话可以快速接手。
8. 运行合适的格式、类型检查、测试或构建验证；如果未运行，说明原因。

## 10. 与 `INSTRUCTIONS.md` 的关系

`INSTRUCTIONS.md` 是每次上下文都应遵守的高频开发协议。

本文件只在用户要求文档构建、文档整理或项目知识库维护时使用，负责提供详细的前端文档体系和写作规则。

如果两者存在冲突，你应该优先遵守 `INSTRUCTIONS.md` 中的高优先级开发安全规则，同时将本文件作为文档任务的补充约束。
